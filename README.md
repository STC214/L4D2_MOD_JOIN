# L4D2 Mod Join V2

Windows 下的 Left 4 Dead 2 VPK 分类合并、冲突处理与安全部署工具。

程序会扫描用户选择目录中的全部源 VPK，根据包内实际资源路径动态分类，
不依赖固定的 Steam Workshop ID 清单。合并前集中处理冲突，合并后校验产物，
最后部署到游戏 `addons` 并禁用被替代的原 MOD。

## 主要功能

### 动态扫描与分类

- 扫描来源目录中的全部 VPK 主包；
- 自动识别 UI/HUD、生还者、感染者、武器、环境、特效、音效、游戏脚本、
  喷漆、地图和其他资源；
- 无法明确识别的 MOD 放入 `11_Misc.vpk`，不会静默丢弃；
- 扫描结果保存为 EXE 同级目录中的 `mod-scan-report.json`；
- 使用源包指纹检测扫描后的文件变化，过期方案不能继续合并或部署。

### 冲突处理

- 同路径、同内容的文件自动去重；
- 可安全组合的纯 `IncludeScript()` 入口和简单喷漆 manifest 自动合并；
- 明确的完整资源覆盖关系自动选择包含更完整的一方；
- 其他冲突按参与竞争的 MOD 集合聚合，在统一窗口中批量处理；
- 冲突窗口预选推荐项并显示理由，也允许逐组修改；
- 复杂脚本或配置冲突不会被程序擅自拼接，必须明确选择保留来源。

### 可靠合并

- VPK 先写入临时文件，完成结构与 CRC 校验后再原子替换正式文件；
- 分类输出之间会再次检查重复运行时路径；
- 成功构建后在 EXE 同级目录生成 `l4d2modjoin-build.json`；
- 部署前重新校验来源指纹、冲突策略和全部输出文件 SHA-256；
- 新构建产生的过期合并包会归档，不会与当前产物混放。

### 安全部署与还原

- 部署前要求 Left 4 Dead 2 已完全退出；
- 所有输出先暂存并校验，通过后才统一更新游戏目录；
- 自动备份 `left4dead2\addonlist.txt`；
- 禁用参与合并的 Workshop 源 MOD，只启用当前合并包；
- 扫描 `addons` 根目录，通过完整包指纹和运行时内容签名识别改名复制或
  非订阅的重复 VPK，并一并禁用；
- 不修改内容不重复、未参与本次合并的本地 MOD；
- 旧合并包移到 `left4dead2\l4d2modjoin_backup`，不会留在 `addons`
  中形成幽灵冲突；
- “一键还原官方MOD”会移出当前合并包，并精确恢复首次部署前保存的
  `addonlist.txt` 基线，包括原本已禁用的条目。
- EXE 同级部署记录可同时保存多个 `addons` 目录的独立状态，切换游戏安装目录
  不会覆盖其他目录的还原信息；
- 部署记录缺失、损坏或不属于当前 `addons` 时会阻止还原，避免只恢复
  `addonlist.txt` 却把合并 VPK 遗留在游戏目录。

部署和还原都不会删除 Workshop 或用户的本地 VPK。备份、旧合并包和被替换文件
均保留在游戏目录外的工具备份文件夹中。

### Win32 界面

- 扫描、合并、部署和还原均在后台执行，不阻塞 UI 消息循环；
- 合并按分类包显示进度；部署按预检、实际复制字节、SHA-256 校验、重复 MOD
  扫描和提交阶段连续更新进度；还原按备份读取、合并包移出和状态恢复更新进度；
- 主窗口和冲突窗口采用内存双缓冲，并避免父窗口擦除子控件区域；
- 日志最新消息显示在顶部；
- 用户向下滚动阅读旧日志时，新消息不会抢走当前阅读位置；
- 执行任务期间禁用操作按钮，避免多个文件任务重叠运行。

## 使用方法

1. 启动 `L4D2ModJoin.exe`。
2. 选择包含待处理 VPK 的“MOD 来源目录”。
3. 选择“合并输出目录”。
4. 确认程序识别出的 `left4dead2\addons`，或手动选择该目录。
5. 点击“智能扫描 MOD”。
6. 查看分类结果和冲突日志。
7. 点击“一键分类合并”。
8. 如果出现冲突窗口，确认推荐方案或逐组调整后开始合并。
9. 完全退出 Left 4 Dead 2。
10. 点击“部署并禁用原 MOD”。
11. 启动游戏，在“附加内容”中确认合并包已启用。

来源目录中的 VPK 在扫描或合并后发生变化时，请重新执行“智能扫描 MOD”和
“一键分类合并”，不要继续使用旧构建清单。

## 自动生成的文件

所有 JSON 状态文件统一保存在 `L4D2ModJoin.exe` 同级目录，不写入游戏
`addons`，也不依赖合并输出目录继续存在：

| 文件 | 用途 |
| --- | --- |
| `mod-scan-report.json` | 扫描、分类和冲突详情 |
| `mod-conflict-policy.json` | 当前来源指纹对应的冲突选择 |
| `l4d2modjoin-build.json` | 可部署文件、来源和 SHA-256 校验清单 |
| `.l4d2modjoin-deployment.json` | 各 `addons` 目录独立的部署位置、源包和已部署合并包清单 |

合并输出目录只保存按实际扫描结果生成的 `01_*.vpk` 至 `11_*.vpk` 分类包。

游戏目录附近只会生成非 JSON 的运行备份：

| 路径 | 用途 |
| --- | --- |
| `addonlist.txt.l4d2modjoin.*.bak` | 部署前的附加内容状态备份 |
| `l4d2modjoin_staging` | 部署过程的临时暂存目录 |
| `l4d2modjoin_backup` | 旧合并包和还原时移出的文件 |

## 构建

构建环境需要：

- Go；
- Python 3 和 Pillow；
- `rsrc` 命令。

在项目根目录运行：

```powershell
.\build.ps1
```

输出文件：

```text
dist\L4D2ModJoin.exe
```

构建脚本会：

1. 使用根目录的 `ico.jpg` 生成多分辨率 `assets\app.ico`；
2. 将应用清单和图标嵌入 EXE；
3. 运行全部 Go 测试；
4. 构建无控制台窗口的单文件 Win32 程序。

`L4D2ModJoin.exe` 运行时不依赖 Python、Go、`rsrc` 或外部图标文件。

## 安全边界

- 工具只处理 VPK、自己的部署清单和 `addonlist.txt`；
- 不注入游戏进程，不读取游戏内存，不模拟输入；
- 游戏运行时会阻止部署和还原；
- 无法读取的无关本地 VPK 不会阻止部署，也不会被自动禁用；
- 只有与本次源包内容指纹一致的本地 VPK 才会被判断为重复 MOD。
- 迁移旧输出目录中的 JSON 时，只有结构有效且指纹与 EXE 同级状态一致的
  工具文件才会删除；同名的用户 JSON 会保留。
